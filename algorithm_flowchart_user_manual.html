1. Introduction

An algorithm is a finite, ordered set of instructions used to solve a problem. A flowchart is a graphical representation of those instructions, showing the control flow from one step to the next.

Flowcharts are important because they simplify complex logic, help identify errors early, and improve communication between developers, students, and stakeholders.

Advantages of Algorithms and Flowcharts

  • Improve clarity before coding begins.
  • Encourage systematic and modular problem-solving.
  • Make debugging and maintenance easier.
  • Serve as documentation for learning and collaboration.

3. Algorithms and Formulae Used in This Project

This section includes only the algorithms that are actually used in the AI interview platform codebase.

3.1 Response Scoring Pipeline

Used in: src/utils/aiAnalysis.js (analyzeResponse)

Purpose: Score each candidate answer using keyword relevance, confidence, and weighted aggregation.

Core Formulae

keywordScore = round((matchedKeywords / expectedKeywords) * 100)
accuracy = round(keywordScore * 0.6 + relevance * 0.4)
overallScore = round((relevance * 0.25 + accuracy * 0.30 + confidence * 0.20 + keywordScore * 0.25) * weight)
overallScore = clamp(overallScore, 0, 100)

Pseudocode

ANALYZE_RESPONSE(answer, question, expectedKeywords, weight)
  keywordScore = keyword_match(answer, expectedKeywords)
  skills = detect_skills(answer)
  confidence = confidence_heuristic(answer)
  relevance = relevance_overlap(answer, question)
  accuracy = round(keywordScore * 0.6 + relevance * 0.4)
  overall = round((relevance*0.25 + accuracy*0.30 + confidence*0.20 + keywordScore*0.25) * weight)
  return clamp(overall, 0, 100), relevance, accuracy, confidence, skills

3.2 Interview Summary Aggregation

Used in: src/utils/aiAnalysis.js (generateInterviewSummary)

Purpose: Compute overall interview metrics from per-question analysis results.

Core Formulae

avgOverall = round(sum(question.overallScore) / totalQuestions)
avgRelevance = round(sum(question.relevance) / totalQuestions)
avgAccuracy = round(sum(question.accuracy) / totalQuestions)
avgConfidence = round(sum(question.confidence) / totalQuestions)
skillAvg(skill) = round(sum(skillScores) / count(skillScores))

Pseudocode

GENERATE_SUMMARY(questionResults)
  if empty -> return zeros
  compute averages with reduce
  aggregate skill scores per skill key
  average each skill bucket
  count strengths and improvements
  sort by frequency and take top items
  return summary object

3.3 Candidate Ranking

Used in: src/utils/aiAnalysis.js (rankCandidates)

Purpose: Rank candidates by score for interviewer dashboards.

Core Formula

rank = position_in_descending_sort(overallScore) + 1

Pseudocode

RANK_CANDIDATES(candidates)
  sorted = sort candidates by overallScore descending
  for each candidate at index i:
    candidate.rank = i + 1
  return sorted

3.4 Job Description to Question Generation

Used in: src/utils/questionGenerator.js (generateQuestionsFromJobDescription)

Purpose: Convert job description text into structured technical, behavioral, and situational interview questions.

Method

  • Extract skills using regex-based technology patterns.
  • Extract responsibilities from bullets/verb-based lines.
  • Generate question templates using role + skill context.
  • Normalize fields and remove duplicate questions.

Pseudocode

GENERATE_QUESTIONS(jobDescription)
  role = extract_role_title(jobDescription)
  skills = extract_skills_with_regex(jobDescription)
  responsibilities = extract_bullets(jobDescription)
  draft = build_template_questions(role, skills, responsibilities)
  normalized = normalize_question_fields(draft)
  return unique_by_text(normalized)

5. Technologies Used

This section lists the technologies used in this project and explains why each technology was selected and how it contributes to the complete system.

Frontend: HTML5, CSS3, JavaScript (ES6+)

Why chosen: These core web standards are stable, universally supported, and ideal for building accessible interfaces.

Contribution: They provide semantic page structure, responsive styling, and client-side interaction logic throughout the app and this manual.

Frontend Framework: React 18 + React Router DOM

Why chosen: React enables reusable component-based development, while React Router handles multi-page SPA navigation cleanly.

Contribution: Powers candidate and interviewer portals, route-based page transitions, and stateful UI modules such as interview flows and dashboards.

UI and Visualization: Tailwind CSS, Lucide React, Recharts

Why chosen: Tailwind accelerates consistent styling, Lucide provides scalable iconography, and Recharts simplifies chart rendering.

Contribution: Delivers cohesive design, modern visual components, and data-driven dashboards for interviewer analytics.

Backend Runtime: Node.js (v18+)

Why chosen: Node.js is efficient for JSON APIs, uses the same language as frontend logic, and has strong package ecosystem support.

Contribution: Runs local backend services, processes session/candidate operations, and supports utility endpoints.

Backend/API Layer: Vercel Serverless Functions

Why chosen: Serverless functions reduce infrastructure overhead and integrate directly with Vercel deployment.

Contribution: Hosts API routes under api/ for sessions, candidates, and health checks in production.

Backend Scripting: Python

Why chosen: Python is strong for automation and document-generation workflows.

Contribution: Executes helper scripts for PPT creation, inspection, and symbol updates used in supporting workflows.

Databases and Persistence: localStorage, JSON (db.json), Vercel KV

Why chosen: localStorage and JSON files are simple for development; Vercel KV adds cloud persistence for deployed serverless APIs.

Contribution: Stores sessions/candidates locally during development and supports persistent shared interview data in deployment.

AI/ML and Computer Vision: face-api.js + AI analysis utilities

Why chosen: face-api.js provides browser-based facial analysis capabilities; custom analysis utilities tailor scoring to interview needs.

Contribution: Enables avatar-related intelligence and automated response analysis features in the interview process.

Web APIs: Web Speech API, MediaRecorder API

Why chosen: Native browser APIs reduce external dependencies and provide direct speech/recording access.

Contribution: Supports speech recognition/transcription and audio-video recording during candidate interviews.

APIs and Third-Party Tools: Vite, ESLint, PostCSS, Autoprefixer, uuid, date-fns, mkcert, Vercel

Why chosen: These tools improve build speed, code quality, CSS compatibility, unique ID generation, date handling, local HTTPS support, and deployment.

Contribution: They streamline development, maintain consistency, and provide reliable production hosting and API delivery.

6. APIs Used (Function and Why)

This project uses both backend HTTP APIs and browser APIs. The table below explains each API, what function it performs, and why it is used.

6.1 Backend HTTP APIs (Vercel/Node)

Endpoint Methods Function Why Used in Project
/api/sessions GET, POST, PATCH, DELETE Create, read, update, and delete interview sessions. Needed for interviewer session lifecycle management and editable interview setup.

/api/health GET Return service health and storage mode (KV or memory). Used for operational checks and deployment diagnostics.
